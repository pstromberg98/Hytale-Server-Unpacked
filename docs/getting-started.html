Getting Started

Everything you need to create your first Hytale server plugin, from project setup to deployment.

Prerequisites

Before you begin developing Hytale server plugins, make sure you have the following installed and configured:

Requirement Minimum Version Notes
Java Development Kit (JDK) 21+ The server is compiled with Java 21. Use Eclipse Adoptium or Oracle JDK.
Gradle 8.x Build tool for compiling and packaging plugins. Use the Gradle wrapper (gradlew) included in the project.
Java Knowledge — Familiarity with Java classes, interfaces, generics, and lambda expressions.
IDE (recommended) — IntelliJ IDEA or Eclipse with Gradle integration for autocompletion and debugging.

Verify your Java installation by running:

java -version
# Expected output: openjdk version "21.x.x" or higher
Warning

Java 21 is the minimum required version. Plugins compiled with a higher JDK version may not load on servers running JDK 21. Set your Gradle toolchain to target Java 21 to ensure compatibility.

Project Setup

Hytale server plugins are standard Java projects built with Gradle. The server JAR is added as a compileOnly dependency so your plugin can reference server APIs without bundling them.

build.gradle.kts

Create a build.gradle.kts file in your plugin project root:

plugins { java }

group = "com.example.myplugin"
version = "1.0.0"

repositories {
    mavenCentral()
    maven { url = uri("https://maven.hytale.com/release") }
}

dependencies {
    // Server API - compileOnly so it is NOT bundled into your JAR
    compileOnly("com.hypixel.hytale:Server:2026.01.24-6e2d4fc36")

    // Optional: shared core library if your project has one
    implementation(project(":libraries:core"))
}

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)
    }
}

tasks.jar {
    archiveBaseName.set("MyPlugin")
    archiveVersion.set("")
    from(configurations.runtimeClasspath.get().map {
        if (it.isDirectory) it else zipTree(it)
    })
    duplicatesStrategy = DuplicatesStrategy.EXCLUDE
}

Directory Structure

A minimal plugin project follows the standard Gradle/Java directory layout:

my-plugin/ ├── build.gradle.kts ├── settings.gradle.kts └── src/ └── main/ ├── java/ │ └── com/example/myplugin/ │ └── MyPlugin.java └── resources/ └── manifest.json
Tip

If your plugin is part of a multi-module Gradle project, add include("plugins:myplugin") to the root settings.gradle.kts.

Your First Plugin

Here is a complete "Hello World" plugin that logs a message on startup and greets players when they connect:

package com.example.myplugin;

import com.hypixel.hytale.server.core.plugin.JavaPlugin;
import com.hypixel.hytale.server.core.plugin.JavaPluginInit;
import com.hypixel.hytale.server.core.event.events.PlayerConnectEvent;
import com.hypixel.hytale.server.core.universe.player.PlayerRef;
import com.hypixel.hytale.server.core.chat.Message;
import javax.annotation.Nonnull;

public class MyPlugin extends JavaPlugin {

    public MyPlugin(@Nonnull JavaPluginInit init) {
        super(init);
    }

    @Override
    protected void setup() {
        getLogger().info("MyPlugin is setting up!");

        // Register a global event listener for player connections
        getEventRegistry().registerGlobal(
            PlayerConnectEvent.class,
            this::onPlayerConnect
        );
    }

    @Override
    protected void start() {
        getLogger().info("MyPlugin is now enabled!");
    }

    @Override
    protected void shutdown() {
        getLogger().info("MyPlugin is shutting down.");
    }

    private void onPlayerConnect(PlayerConnectEvent event) {
        PlayerRef player = event.getPlayerRef();
        player.sendMessage(Message.raw("Welcome to the server!"));
    }
}
Note

Every plugin must extend JavaPlugin and provide a constructor that takes a JavaPluginInit parameter and passes it to super(). The server instantiates your plugin via this constructor.

Plugin Manifest

Every plugin JAR must contain a manifest.json file at src/main/resources/manifest.json. This file tells the server how to identify and load your plugin.

{
  "Group": "com.example",
  "Name": "MyPlugin",
  "Version": "1.0.0",
  "Main": "com.example.myplugin.MyPlugin",
  "Dependencies": []
}

Manifest Fields

Field Type Required Description
Group String Required The group/organization identifier, typically a reversed domain name (e.g. "com.example"). Combined with Name to form the unique plugin identifier.
Name String Required The plugin name. Must be unique within the server. Used for logging and identification.
Version String Required Semantic version string (e.g. "1.0.0"). Displayed in server logs on load.
Main String Required The fully-qualified class name of your main plugin class that extends JavaPlugin. The server uses this to instantiate your plugin.
Dependencies Array Optional List of plugin identifiers ("group:name") that must be loaded before this plugin. The server resolves load order from this field.
Warning

The Main class must have a public constructor accepting a single JavaPluginInit parameter. If the constructor is missing or has the wrong signature, the server will fail to load the plugin with a reflection error.

Plugin Lifecycle

The server manages plugin state through a defined lifecycle. Your plugin class can override three hooks that are called in order:

Constructor
→
setup()
→
start()
→
Running
→
shutdown()

setup()

Called after the constructor, during the SETUP phase. This is where you should register all event handlers, commands, configs, and other registries. The server is not yet fully started at this point.

@Override
protected void setup() {
    // Register events
    getEventRegistry().registerGlobal(PlayerConnectEvent.class, this::onConnect);
    getEventRegistry().registerSystem(KillFeedEvent.Display.class, this::onDeath);

    // Register commands
    getCommandRegistry().register(new SpawnCommand());

    // Register configs (must be done before setup completes)
    withConfig(MyConfig.CODEC);
}

start()

Called after setup() completes and the plugin transitions to the START phase. At this point all registries are active and the server is running. Use this for initialization logic that depends on other plugins or server state being available.

@Override
protected void start() {
    getLogger().info("Plugin enabled with " +
        Universe.get().getWorlds().size() + " worlds loaded");
}

shutdown()

Called when the server is stopping or the plugin is being disabled. Use this for cleanup: save data, close connections, cancel tasks. After this method returns, all registries (events, commands, tasks) are automatically cleaned up by the server.

@Override
protected void shutdown() {
    getLogger().info("Saving data and cleaning up...");
    // Save any persistent data here
}
Tip

You do not need to manually unregister events or commands in shutdown(). The server automatically cleans up all registrations made through getEventRegistry(), getCommandRegistry(), and other plugin registries when the plugin is disabled.

Critical

If setup() throws an exception, the plugin state is set to DISABLED and start() will never be called. Always wrap risky initialization in try-catch blocks and log failures clearly.

Core APIs Quick Reference

This table shows the most common operations you will need when developing plugins. Each entry links to the relevant documentation page for full details.

Task Code Details
Get Universe Universe.get() Singleton root — access worlds, players, and global state. Worlds & Universe
Get World Universe.get().getWorlds().get("name") Returns a World by its string key. Returns null if not found. Worlds & Universe
Register Event eventRegistry.register(EventClass.class, handler) Use registerGlobal() for IBaseEvent, registerSystem() for EcsEvent. Event System
Register Command commandRegistry.register(cmd) Register a CommandBase subclass. Command System
Send Message playerRef.sendMessage(Message.raw("text")) Send chat message to a specific player. Thread-safe. Players
Schedule on World Thread world.execute(() -> { ... }) Required for EntityStore operations from non-world threads. Threading Model
Get Player Component store.getComponent(ref, Player.getComponentType()) Access full Player component from entity ref. Must be on world thread. ECS Architecture
Warning

Most Store operations (getting components, adding entities) must be called from the world thread. Use world.execute(() -> { ... }) to schedule work on the correct thread from commands or async contexts. See the Threading Model documentation for full details.

Project Structure

A well-organized plugin project follows this standard layout. Larger plugins should separate concerns into subpackages:

my-plugin/ ├── build.gradle.kts # Gradle build config ├── settings.gradle.kts # Gradle settings (multi-module) └── src/ └── main/ ├── java/ │ └── com/example/myplugin/ │ ├── MyPlugin.java # Main entry point (extends JavaPlugin) │ ├── commands/ │ │ ├── SpawnCommand.java │ │ └── TeleportCommand.java │ ├── events/ │ │ ├── JoinListener.java │ │ └── DeathListener.java │ └── util/ │ └── MessageUtil.java ├── resources/ │ └── manifest.json # Plugin manifest (REQUIRED) └── assets/ # Data mod assets (optional) ├── manifest.json # Data mod manifest (separate from plugin manifest) └── Server/ ├── Item/Items/... ├── SoundEvent/... └── Audio/...
Note

The src/main/assets/ directory is only needed if your plugin includes a data mod (custom items, sounds, etc.). The data mod manifest inside assets/manifest.json is separate from the plugin manifest in resources/manifest.json. See Plugin Development for data mod details.

Building & Deploying

Building

Use Gradle to compile and package your plugin into a JAR:

# Build all plugins in the project
./gradlew build

# Build a specific plugin module
./gradlew plugins:myplugin:jar

The compiled JAR will be placed in:

plugins/myplugin/build/libs/MyPlugin.jar

Deploying

To install your plugin on a Hytale server, copy the compiled JAR into the server's mods/ directory:

# Copy the JAR to the server's mods folder
cp plugins/myplugin/build/libs/MyPlugin.jar /path/to/server/mods/

# Restart the server to load the plugin
# The server scans /mods on startup for plugin JARs
Warning

Plugins are loaded once at server startup. Hot-reloading of plugins is not supported. You must restart the server after adding, removing, or updating a plugin JAR.

Automated Deployment

If your project includes deployment scripts, you can use them for remote deployment:

# Deploy to a remote server
./scripts/remote-deploy.sh user@server.local

# Deploy without restarting the server
./scripts/remote-deploy.sh user@server.local --no-restart

Verifying Plugin Load

After starting the server, check the console logs to verify your plugin loaded correctly:

# Look for these messages in the server log:
[INFO] [com.example:MyPlugin] MyPlugin is setting up!
[INFO] [com.example:MyPlugin] MyPlugin is now enabled!

If your plugin fails to load, the server will log the error with a stack trace. Common issues include:

  • Missing or malformed manifest.json
  • Wrong Main class path in the manifest
  • Missing public constructor with JavaPluginInit parameter
  • Compile-time dependency not available at runtime
